维护者入门简介
========================

维护者入门简介补充了顶级流程文档（如提交补丁、提交驱动程序等）中的子系统/设备驱动器本地习俗以及补丁提交生命周期的详细信息。贡献者使用此文档来调整他们的期望并避免常见错误；维护者可以使用这些简介跨子系统寻找统一常用实践的机会。

概述
--------
介绍子系统的运作方式。虽然MAINTAINERS告诉贡献者应将哪些补丁发送到哪些文件，但它没有传达其他帮助开发的子系统本地基础设施和机制。
考虑的问题示例：

- 当补丁被应用到本地树或合并到上游时，是否有通知？
- 子系统是否有Patchwork实例？Patchwork状态更改是否被通知？
- 是否有监控邮件列表的机器人或持续集成（CI）基础设施，或者子系统是否使用自动测试反馈作为接受补丁的门槛？
- 被拉入-next的Git分支有哪些？
- 贡献者应针对哪个分支提交？
- 任何其他维护者入门简介的链接？例如，设备驱动程序可以指向其父子系统的条目。这使贡献者意识到在提交链中对其他维护者的义务。

提交检查清单附加项
-------------------------
列出超出通用“提交检查清单”的强制性和建议性标准，以使补丁足够健康以引起维护者的注意。
例如：“通过checkpatch.pl且无错误或警告。通过$URI中详述的单元测试。”
提交检查清单附加项还可以包括有关相关硬件规范状态的详细信息。例如，子系统是否要求在考虑补丁之前发布特定修订版本的规范？

关键周期日期
---------------
提交者常见的误解之一是可以在合并窗口关闭前的任何时候发送补丁，并且仍然可以考虑用于下一个-rc1。实际情况是大多数补丁需要提前在linux-next中稳定下来，才能在合并窗口打开时进行合并。为提交者澄清关键日期（以-rc发布周为准），补丁可能被考虑合并的时间点以及补丁需要等待下一个-rc的时间点。至少包括：

- 最后一个-rc用于新功能提交：
  针对下一个合并窗口的新功能提交应在这一点之前进行首次发布以供考虑。在此之后提交的补丁应明确指出它们的目标是NEXT+1合并窗口，或应附上足够的理由说明为何应加速考虑。一般指导原则是告知贡献者新功能提交应在-rc5之前出现。
- 最后一个-rc用于合并功能：合并决策的截止日期
  告知贡献者，在这一点之前尚未应用的补丁集将需要等待NEXT+1合并窗口。当然，没有任何义务必须接受任何一个给定的补丁集，但如果在此时审查尚未结束，则预期贡献者应等待并在下一个合并窗口重新提交。

可选：

- 开发基线分支（在概述部分列出）应被认为准备好接受新提交的第一个-rc。

审核节奏
--------------
导致贡献者焦虑的最大来源之一是在发布补丁集后多久未收到任何反馈就该发出提醒。除了指定重新提交前要等待的时间外，本节还可以指示首选的更新方式，如重新发送整个系列或私下发送提醒邮件。本节还可能列出此代码区域的审核方式及获取反馈的方法，而不仅仅是直接从维护者那里获得反馈。
